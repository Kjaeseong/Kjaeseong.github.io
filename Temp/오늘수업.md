#### crypto

#### 단방향 암호화
- 복호화할 수 없는 암호화 방식
  - 복호화 : 암호화된 문자열을 원래 문자열로 되돌려놓는 것
- 주로 해시 기법을 사용
  - 해시 함수라고 부르기도 한다
- 유저의 비밀번호 자체는 저장하지 않고, 암호화된 문자열로만 비교, 데이터베이스의 비밀번호와 비교
- createHash(알고리즘) : 
  - 사용할 해시 알고리즘.
  - md5, sha1, sha256, sha512등 가능
    - md5, sha1은 취약점이 발견
  - 추후 sha512마저 취약해지면 더 강화된 알고리즘으로 바꿔야 한다.
- update(문자열) : 변환할 문자열
- digest(인코딩) :
  - 인코딩할 알고리즘. 변환된 문자열을 반환
  - base64, hex, latin1 주로 사용
  - base64가 결과 문자열이 가장 짧아 애용

#### 양반향 암호화
- 암호화된 문자열 복호화 가능
- '키' 사용.
- 복호화 시 암호화할 때 사용한 키와 같은 키를 사용해야 한다.
- 완벽한 이해를 위해 암호학을 추가로 공부해야 한다.

```javascript
const crypto = require('crypto');

const algorithm = 'aes-256-cbc';
const key = 'abcdefghijklmnopqrstuvwxyz123456';
const iv = '1234567890123456';

// 암호화
const cipher = crypto.createCipheriv(algorithm, key, iv);
let result = cipher.update('암호화할 문장', 'utf8', 'base64');
result += cipher.final('base64');
console.log('암호화', result);

// 복호화
const decipher = crypto.createDecipheriv(algorithm, key, iv);
let result2 = decipher.update(result, 'base64', 'utf8');
console.log('복호화', result2);
```

- localhost : 현재 컴퓨터의 내부 주소. 자신의 컴퓨터에서만 접근할 수 있다. 서버 개발시 테스트용으로 많이 사용

// server1-1.js

const http = require('http');

const server = http.createServer((req, res) => {
    res.writeHead(200, { 'content-Type': 'text/html; charset=utf-8' });
    res.write('<h1>hello Node!</h1>');
    res.end('<p>Hello Server!</p>');
});
server.listen(8080);

server.on('listening', () => {
    console.log('8080 포트에서 서버 대기 중');
});
server.on('error', (error) => {
    console.error(error);
});

### 게임 서버의 종류
- 서버 : 클라이언트들이 참가할 수 있는 네트워크 공간을 마련하는 컴퓨터(= 호스트)

#### 전용 서버
- 서버의 모든 자원이 온전히 네트워크 서비스를 유지하는 데 사용
- 서버가 플레이어로서 게임에 직접 참가하지 않는다.
- 고정비용 많이 발생


#### 리슨 서버
- 전용 서버가 없는 대신 플레이어 클라이언트 중 하나가 서버 역할을 맡는다.
- Play as Host 라고 부르기도 한다.
- 서비스 유지비용이 적다.
- 호스트의 컴퓨터 성능에 따라 네트워크 품질이 크게 달라진다.
- 호스트의 종료시 진행중인 게임을 정지하고 남은 클라이언트 중 새로운 호스트를 선정하는 절차 필요

#### P2P
- 게임에 참가한 클라이언트들 모두가 호스트 역할을 겸함
- 호스트에 연결된 구조가 아니라 서로 직접 연결된 형태
- 연산 대부분을 독점하는 특정 호스트가 없다
- 각자 자신의 월드에서 담당 연산 실행. 다른 클라이언트에 결과 전파
- 호스트가 게임 도중 접속 종료시, 호스트 교체 과정 불필요
- 서버 유지비용이 없다
- 프로그램의 처리 흐름을 직관적으로 짤 수 있다
- 참가자가 증가할수록 반응 속도가 느려진다.
  - 각 클라이언트가 부담하게 되는 연산량이 늘어난다.
  - 통상 16명을 참가자 상한선으로 여긴다.
  - 16명도 많은 편.
- 클라이언트는 해킹에 취약하다
  - P2P 방식은 수치 변조에 제일 취약하다.

#### 매치메이킹 서버
- 매치메이킹 과정에서 포톤에서 제공하는 전용 클라우드 서버 사용
- 매치메이킹 서버 : 
  - P2P나 리슨 서버라도 클라이언트들이 서로를 찾아 방 하나에 모이는 과정에서 사용할 전용 서버
- 룸이 구성되고 라운드를 시작할 시 플레이어중 한 명을 호스트로 삼는 리슨서버 방식 많이 사용

#### 포톤 룸

    - 유니티와는 따로 동작한다
  - 같은 룸에 있지만 서로 다른 씬을 로드하는것도 가능

### 네트워크 권한 분리
- 공정한 게임 결과를 보장하고 수치에 대한 위변조를 방지해야 한다.

#### 호스트에 위임
- 이유
  - 동기화에 오차가 존재하는 경우 기준이 되는 월드를 정하기 위해
  - 클라이언트의 변조나 위조 행위를 막기 위해
- 네트워크 동기화는 전송 속도나 패킷 손실 등의 현실적인 문제 때문에 완벽하지 않다.
- 게임 승패와 직접적인 관려이 있는 중요한 처리는 호스트에 위임해야 한다.
  - 클라이언트는 호스트의 결과를 받고 시각적인 처리만 수행한다.

#### RPC
- 어떤 메서드나 처리를 네트워크를 넘어 다른 클라이언트에서 실행하는 것
- 대부분의 멀티플레이어 API에는 RPC가 구현되어 있다.
