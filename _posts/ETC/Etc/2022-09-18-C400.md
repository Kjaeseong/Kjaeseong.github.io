# 소프트웨어 구조 다이어그램과 C4 모델
- [C4모델 페이지](https://c4model.com/)

## 소프트웨어 구조 다이어그램
- 소프트웨어의 기능과 구조, 객체들간의 관계를 그림, 글로 작성한 것.
- 소프트웨어 구조 파악시 글보다 그림을 통해 빠른 설명과 이해가 가능하다.
- 작성 시 모델링과 일러스트레이팅 영역을 구분하는게 좋다.
  - `모델링` : 특정 모델(이론) 기반, 소프트웨어를 이루는 각 요소가 어떤 관계를 맺고 있는지 파악
  - `일러스트레이팅` : 모델링으로 파악한 정보를 시각적으로 표현
  - 협업이 쉬워진다
  - 변경 사항을 쉽게 동기화 할 수 있는 구조가 된다
  - 컨텐츠에 집중할 수 있다
  - 텍스트 기반으로 작업 할 수 있다.
  - 구조 모델을 데이터로 관리할 수 있다.

### 작성 시 지켜야 할 점
- 다이어그램을 작성, 유지보수에 비용과 시간이 적어야 한다
- 일관된 구조 뷰와 정확한 표기 기준에 맞춰 작성해야 한다.
- 개발 관계자라면 누구나 이해하고 그릴 수 있도록 복잡하거나 어렵지 않아야 한다.

## C4 모델
- UML과 4+1 아키텍처 뷰 모델을 기반으로 만든 모델
- 소프트웨어를 작은 단위로 분해해가는 방식으로 모델링하는 기법
  - 시스템 콘텍스트 > 컨테이너 > 컴포넌트 > 코드
- 개발을 빠르게 진행하는 상황에 소프트웨어의 구조를 빠르고 효율적으로 공유, 업데이트시 사용 권장
- Static View 소프트웨어 구조를 나타내는 데 적합하다.

### 추상화 대상

| 추상화 대상           | 설명                                                                                                                                                  |
|------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------|
| 사용자<br/>(Person) | 사용자나 역할 등을 표현<br/>소프트웨어 시스템의 사용자를 나타냄                                                                                                               |
| 소프트웨어 시스템        | 최상위 추상화 요소로 시스템이 어떤 가치를 제공하는지 나타냄<br/>보유, 개발중인 소프트웨어와 연동되는 소프트웨어를 나타낼 때 사용                                                                          |
| 컨테이너             | 소프트웨어시스템의 내부를 표현하는 추상화 요소<br/>데이터 저장과 관련된 솔루션 표현시 사용<br/>서버, 클라이언트 앱, CLI 앱 또는 배치 프로세스, 데이터베이스, 파일 시스템 등과 같은 것을 나타냄<br/>독립적으로 배포할 수 있는 소프트웨어 단위로 구분 |
| 컴포넌트             | 컨테이너 내부를 표현하는 추상화 요소<br/>기능 단위로 묶을 수 있는 모듈, 인터페이스의 집합 표현시 사용<br/>독립적으로 배포할 수 있는 소프트웨어 단위가 아닌 것으로 구분<br/><br/>Ex> 인터페이스나 패키지를 구현하기 위해 구현한 클래스의 집합    |
| 관계               | 추상화 요소 사이의 의존성, 데이터 흐름 등 표현                                                                                                                         |

### 가이드라인

| 표기 가이드라인 | 설명                                                                               |
|----------|----------------------------------------------------------------------------------|
| 다이어그램    | 제목을 표시하고 수준을 알 수 있게 작성<br/>되도록 다이어그램의 범례 표시<br/>약어나 두문자어는 독자가 이해할 수 있도록 표기       |
| 요소       | 모든 요소는 타입이 무엇인지 명확히 구분<br/>모든 요소에 짧은 설명 첨부<br/>컨테이너, 컴포넌트는 사용, 혹은 기반 기술이 무엇인지 표기 |
| 관계       | 되도록 단방향 선으로 관계 표시<br/>모든 선에 설명 작성<br/>컨테이너 간 데이터를 주고받는 기술, 프로토콜을 표기              |
